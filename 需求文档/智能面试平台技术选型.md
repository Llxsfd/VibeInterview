# 智能面试与计算机基础知识复习平台技术选型文档

## 1. 文档说明

### 1.1 文档目的

本文档用于说明“智能面试与计算机基础知识复习平台”的技术选型方案，为后续系统架构设计、数据库设计、接口开发、模型接入、部署上线和后期扩展提供技术依据。

本项目当前版本定位为：

```text
用户上传 PDF 学习资料
  → 系统解析并构建个人知识库
  → 用户基于资料进行智能问答
  → 系统生成题目和模拟面试
  → 用户文字或语音回答
  → AI 自动评分和生成报告
```

当前阶段暂不设计独立管理端，系统重点面向普通用户自助使用。

---

### 1.2 技术选型原则

本项目技术选型遵循以下原则：

```text
1. 优先保证项目可落地。
2. 优先保证 RAG 问答准确性。
3. 优先选择生态成熟、文档完善的技术。
4. MVP 阶段避免过度架构。
5. 保留后期扩展管理端、公共知识库、私有化部署的能力。
6. AI 模块尽量解耦，方便替换模型服务。
7. 文档解析、检索、评分、语音模块应独立封装，便于维护。
```

## 6. 数据库与向量存储选型

## 6.1 主数据库

### 推荐技术

```text
PostgreSQL
```

### 选择原因

```text
1. 适合存储用户、文档、chunk、问答、面试、错题、计划等关系型数据。
2. 支持 JSONB，适合存储 metadata、评分结果、报告内容等半结构化数据。
3. 生态成熟，稳定可靠。
4. 可以通过 pgvector 扩展支持向量检索。
```

---

## 6.2 向量存储

### 推荐技术

```text
pgvector
```

### 选择原因

```text
1. 可以直接在 PostgreSQL 中存储向量。
2. 部署简单，适合 MVP。
3. 用户数据和向量数据可以放在同一个数据库中，权限过滤更容易。
4. 对个人知识库规模足够使用。
5. 后期可以迁移到 Milvus、Qdrant 等专业向量数据库。
```

---

## 6.3 为什么 MVP 不直接使用 Milvus

MVP 阶段不建议直接使用 Milvus，原因是：

```text
1. 当前数据规模初期不会很大。
2. PostgreSQL + pgvector 已经可以满足个人知识库检索。
3. Milvus 会增加部署和维护复杂度。
4. 当前阶段更重要的是 PDF 入库质量和 RAG 检索链路，而不是大规模向量检索。
```

### 适合升级到 Milvus 的情况

```text
1. chunk 数量达到百万级以上。
2. 用户并发检索量明显升高。
3. 向量检索性能成为瓶颈。
4. 需要独立扩展向量检索服务。
5. 需要构建平台级公共知识库。
```

## 7.2 异步任务

### 推荐技术

```text
Celery + Redis
```

### 选择原因

PDF 解析、OCR、embedding、自动出题、报告生成都可能耗时较长，不适合在 HTTP 请求中同步执行。

### 使用场景

```text

2. OCR 识别任务。
3. 文本清洗和 chunk 切分任务。
4. embedding 向量化任务。
5. 知识点自动提取任务。
6. 题目自动生成任务。
7. 面试报告生成任务。
```

### 典型流程

```text
用户上传 PDF
  ↓
后端保存文件并创建 document 记录
  ↓
投递 Celery 异步任务
  ↓
立即返回上传成功
  ↓
前端轮询文档状态
  ↓
任务完成后状态变为 indexed
```

## 9. PDF 解析技术选型

## 9.1 普通文本 PDF

### 推荐技术

```text
PyMuPDF + pdfplumber
```

### PyMuPDF 适合场景

```text
1. 快速按页提取文本。
2. 获取页码。
3. PDF 转图片。
4. 提取图片。
5. 性能较好。
```

### pdfplumber 适合场景

```text
1. 表格提取。
2. 保留文本布局。
3. 分析字符坐标。
4. 处理排版复杂的 PDF。
```

---

## 9.2 扫描版 PDF

### 推荐技术

```text
PaddleOCR
```

### 使用场景

```text
1. PDF 页面没有可提取文本。
2. PDF 是扫描图片。
3. 用户上传的是拍照版资料。
4. 文本提取结果过短，需要 OCR 补充。
```

---

## 9.3 PDF 解析策略

建议采用混合解析策略：

```text
1. 优先使用 PyMuPDF 按页提取文本。
2. 如果某页文本长度过短，则判断为可能需要 OCR。
3. 将该页 PDF 转换为图片。
4. 使用 PaddleOCR 识别文字。
5. 将普通文本页和 OCR 页统一转换为页面文本结构。
```

### 输出格式

```json
[
  {
    "page": 1,
    "text": "第一章 操作系统概述..."
  },
  {
    "page": 2,
    "text": "1.1 进程与线程..."
  }
]
```

## 11. RAG 技术选型

## 11.1 RAG 实现方式

### 推荐方案

```text
自研轻量 RAG 主流程
```

### 原因

本项目对以下能力要求较高：

```text
1. 用户权限隔离。
2. 文档来源引用。
3. chunk metadata 控制。
4. 混合检索。
5. rerank。
6. 检索日志。
7. 低置信度拒答。
8. 面试题和评分需要关联来源 chunk。
```

这些能力自研主流程更容易控制。

---

## 11.2 是否使用 LlamaIndex

### 可选方案

```text
LlamaIndex 可作为辅助工具，但不建议完全依赖框架黑盒流程。
```

### 使用建议

```text
1. 可以参考 LlamaIndex 的文档索引和检索思想。
2. 可以局部使用其 loader 或 retriever。
3. 核心 chunk 切分、metadata、权限过滤和日志建议自己实现。
```

## 13. Rerank 技术选型

## 13.1 推荐模型

```text
bge-reranker-v2-m3
```

## 13.2 为什么需要 Rerank

向量检索适合粗召回，但排序不一定足够精准。rerank 可以逐个判断用户问题与候选 chunk 的相关性，从而提升 Top 5 结果质量。

### 推荐流程

```text
向量召回 Top 20
关键词召回 Top 20
合并去重
rerank 排序
取 Top 5
```

## 19.3 数据隔离

所有核心表建议都包含：

```text
user_id
```

所有查询必须带上：

```text
WHERE user_id = 当前登录用户 ID
```

这对于文档、chunk、向量、问答、面试、错题和计划都非常重要。

---

## 19.4 文件安全

```text
1. 限制上传文件类型为 PDF。
2. 限制文件大小。
3. 文件路径不要直接暴露服务器真实路径。
4. 用户只能下载或查看自己的文件。
5. 语音文件应支持用户删除。
6. 生产环境必须使用 HTTPS。
```

## 25. 结论

本项目的技术选型应以“快速实现可用闭环”和“保证知识库问答准确率”为核心。

第一阶段不要过度追求复杂架构，建议先完成：

```text
用户注册登录
  → PDF 上传
  → PDF 解析
  → chunk 切分
  → embedding 入库
  → RAG 问答
  → 来源引用
  → 模拟面试
  → AI 评分
```

当核心流程跑通后，再逐步增强：

```text
OCR
rerank
语音回答
错题本
学习计划
管理端
Milvus
Elasticsearch
私有化模型
```

最重要的技术判断是：

```text
MVP 阶段使用 PostgreSQL + pgvector + FastAPI + Next.js。
RAG 主流程尽量自研，保证 chunk、metadata、权限、引用和检索日志可控。
AI 模型和语音服务保持可替换，避免被单一服务绑定。
```
